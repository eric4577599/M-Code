## 2. 發現並修正的瑕疵:CSV 匯出浮點雜訊

### 現象
結果頁 CSV 列出現浮點表示誤差:`overall_score` 顯示 `2.5599999999999`(應 2.56)、`margin_score` `0.5599999999999992`(應 0.56)、`bwr_gain_mm` `0.030000000000000027`(應 0.03)。UI 讀數是乾淨的,但 CSV(ERP 回寫的殺手鐧)原封輸出上游浮點運算的雜訊。

### 根因
`src/export/csv.ts` 的 `escapeField` 對數字直接 `String(value)`,忠實序列化未收斂的浮點值。既有 csv 測試用的是乾淨輸入值,故未覆蓋到此案例。

### 修正(test-safe)
在 csv.ts 加 `formatNumber(n)`:`String(Number(n.toPrecision(10)))` —— **只清除浮點表示誤差,不強加領域精度**。本模型量測/分數皆 <10 位有效數字,合理值原封不動,只收斂尾端雜訊。非有限值(NaN/Inf)回退原樣。`escapeField` 對 `number` 型別改走 `formatNumber`。

- 新增測試「清除浮點表示誤差,不改動合理數值」(tests/csv.test.ts)。
- `npm run typecheck` 乾淨、`npm run build` 乾淨、**測試 125 passed**(原 124 + 1)。

### 驗證(瀏覽器端即時)
用 cache-bust 動態 `import('/dist/export/csv.js?v=…')` 載入新模組,對含雜訊 session 呼叫 `toCsvRow`:
```
2.5599999999999      → 2.56
0.5599999999999992   → 0.56
0.030000000000000027 → 0.03
```
✅ 修正生效。

## 3. 部署層發現:Cloudflare 邊緣快取 .js 達 4 小時

測試中發現 demo 頁改碼後不會立即生效。查對外 `dist/export/csv.js` 標頭:
```
cf-cache-status: HIT
cache-control: max-age=14400   (4 小時)
```
Cloudflare 預設會快取靜態副檔名(.js/.css…)。**影響:更新程式碼後,使用者最長 4 小時才看到新版**(邊緣快取 + 瀏覽器 disk cache 雙層)。

> 本次:邊緣其實已更新(curl 拿到新檔),是瀏覽器 disk cache 認為舊檔仍 fresh。程式與部署皆正確,純快取傳播延遲。

### 建議(擇一,屬 Cloudflare 後台/維運調整,未動手)
1. **(推薦,dev/demo 適用)** Cloudflare Zero Trust/Dashboard → 對 `m-code.ericchh.work` 加一條 **Cache Rule:Bypass cache**。demo 求即時 > 快取。
2. 每次部署後到 Cloudflare **Purge Cache**(手動,單次)。
3. 若要保留快取又要即時:改碼後在 docroot 用帶版本 hash 的檔名(需 build 產出 hash,較大工程)。

### ✅ 已採用解法:原點端 no-store(2026-07-02 實作,免後台)
把 web LaunchAgent 從 `python -m http.server` 改為自訂伺服器 `~/m-code-serve.py`
(仍雙協議綁 `::`),每個回應送 `Cache-Control: no-store, no-cache, must-revalidate, max-age=0`。
- 驗證:對外新資源 `cf-cache-status: BYPASS/DYNAMIC`,Cloudflare 不再快取。
- 既有邊緣快取條目(改碼前)約 2.5h 後自然到期;其內容已是修好的新版,故不影響正確性。若要立即清可到後台 Purge 一次(非必要)。
